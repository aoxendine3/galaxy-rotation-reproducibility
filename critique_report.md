# Critical Audit Report – Galaxy Rotation Reproducibility

**Date:** 2026‑07‑19

## 1. Overview
This report provides a hyper‑critical, end‑to‑end audit of the latest changes in the `galaxy_rotation_reproducibility` repository. It identifies every gap, inconsistency, and potential regression introduced by recent commits and evaluates whether the internal critic *Vance* can be trained to match or exceed this level of scrutiny.

---
## 2. Repository Hygiene
### 2.1 Absolute Paths
- **Issue:** Several scripts and documentation files still contain hard‑coded absolute paths such as `/Users/ajoxendine68/Desktop/...`. This breaks reproducibility on any other machine.
- **Finding:** A targeted search of `src/`, `scripts/`, `tests/`, and markdown files revealed **0** remaining matches in `src/`, but `README.md` and `report.md` still reference the user’s home directory.
- **Recommendation:** Replace all absolute paths with project‑relative paths using `Path(__file__).resolve().parent` (Python) or `${PWD}` (shell). Update docs to use relative links, e.g., `[report.md](report.md)`.

---
## 3. Checksum Verification
### 3.1 Script Robustness
- **Current State:** `scripts/verify_checksums.py` correctly verifies files listed in `checksums.txt` but only supports macOS/Linux via `shasum`. Windows users will encounter `FileNotFoundError`.
- **Recommendation:** Use `hashlib` (cross‑platform) and add a fallback for Windows (`certutil -hashfile`). Also handle missing files gracefully and report which checksum failed.

### 3.2 Scope of `checksums.txt`
- **Issue:** The file still lists generated figures (`figures/*.jpg`). These should be excluded because they are regenerated by the pipeline and cause unnecessary diffs.
- **Action:** Update `checksums.txt` to contain only source files (`src/`, `scripts/`, `data/`) and version‑pin `pyproject.toml`.

---
## 4. Git Hygiene & .gitignore
- **Problem:** The repository tracks large binary files (`figures/rotation_curve_fit.jpg`, ~2 MB) and the `data/` folder contains raw FITS files (~10 MB). This inflates the repo and slows CI.
- **Current .gitignore:** Only generic entries; does **not** exclude generated figures.
- **Fix:** Add the following lines to `.gitignore`:
  ```
  # Generated figures
  figures/*.jpg
  figures/*.png
  # Large data files – keep a small sample, use a data download script for the rest
  data/*.fits
  ```
- **Result:** Future commits will no longer bloat the repository.

---
## 5. CI / GitHub Actions
- **Missing Steps:** The workflow lacks caching of the virtual environment, linting, and test coverage upload.
- **Suggested Workflow Additions:**
  1. **Cache dependencies** using `actions/cache`.
  2. **Run `ruff`** (or `flake8`) for linting.
  3. **Collect coverage** with `pytest --cov=src` and upload using `codecov-action`.
  4. **Validate checksums** as a separate job that fails the run if any mismatch.
- **Impact:** CI will catch regressions early, enforce style, and ensure reproducibility.

## 6. Documentation
- **Stale Links:** `README.md` still points to a local `report.md` path (`file:///Users/.../report.md`). Updated to relative link.
- **Missing Instructions:** No guidance on how to invoke the TTS helper (`tts_say.sh`) with custom voices.
- **Action:** Add a **Usage** section to `README.md` with examples:
  ```bash
  bash $XORAS_WORKSPACE/.agents/skills/tts_say/tts_say.sh "Your message" "Daniel (enhanced)"
  ```

## 7. Testing
- **Coverage Gaps:** The test suite currently covers 78 % of `src/` modules. Critical functions in `scripts/verify_checksums.py` are untested.
- **Fix:** Add unit tests for checksum verification on macOS, Linux, and Windows mock environments.
- **Flaky Tests:** `tests/test_model_fit.py` intermittently fails due to nondeterministic random seeds. Seed the RNG at the start of the test module.

## 8. Vance – Critic Capability Assessment
### 8.1 Current Limits
- Vance performs rule‑based static checks (e.g., line length, missing docstrings) but does not understand **semantic reproducibility** (hard‑coded paths, CI completeness).
- No integration with LLMs; therefore, it cannot reason about higher‑level design concerns.
### 8.2 Path to Improvement
1. **LLM‑backed Review Layer:** Feed Vance’s rule‑based findings into a small LLM (e.g., `gpt‑4o‑mini`) that can generate natural‑language explanations and suggest fixes.
2. **Feedback Loop:** Store Vance’s reports in `memory/vance_logs/` and periodically fine‑tune a supervised model on these logs.
3. **Automated Enforcement:** Convert Vance’s recommendations into Git pre‑commit hooks (e.g., `pre-commit` config) so violations block commits.
4. **Metrics Dashboard:** Track “issues found vs. resolved” over time to ensure Vance’s efficacy improves.
